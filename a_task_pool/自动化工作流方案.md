# 自动化工作流方案：AI-Ready 任务池 × SDD 开发流水线

> 版本：v1.0 · 2026-05-09
> 作者：AI Agent 独立开发（Opus 4.7 协作）
> 定位：把人工审核从"串行瓶颈"拆解为"分类后的旁路"，用结构化任务池驱动 AI 7×24 产出

---

## 0. 核心命题（先说结论）

**问题重述**：现在的瓶颈不是 AI 能力不够，而是"未分类的需求"把所有任务都卡在了人工审核这个木桶短板上。自媒体发稿、GitHub 工具开发、日常数据清洗这类 **人工参与=零增量价值** 的任务，被迫和"必须人工决策"的任务排同一个队。

**解法三段式**：
1. **AI-Ready Data 前置层**：所有任务进池之前，先经过一层结构化契约（Schema + 领域字段 + 人工介入标记），而不是裸奔的自然语言指令。
2. **任务池调度层**：YAML/JSON/Excel 驱动，路由器根据 `human_in_loop` 字段把任务拆成三类通道（全自动 / 事后通知 / 必须人工）。
3. **执行层复用开源**：Claude Code SDD 跑代码类任务，n8n 跑运营/集成类任务，OpenClaw 跑 IM 交互类任务——**不重复造轮子**。

**与 OpenClaw 的关系**（提前回答 task-3）：**不是竞品，是前置协议层**。OpenClaw 解决"agent 怎么常驻并跨 IM 执行"，我们解决"任务怎么被正确定义、分类、路由"。OpenClaw 可以作为本方案的一个执行后端被调用。详见 §7。

## 1. AI-Ready Data 前置范式（最重要，放最前面）

> 这一节直接回应你在 task-4 的强调：**"工作流再好，没有和领域内输入协同，都是拿着锤子找钉子。"**

### 1.1 为什么"AI-Ready"必须前置

2026 年 AI 工作流的共识已经从 "prompt 工程" 迁移到 **"Spec 工程 + 结构化输入"**——见 Spec-Driven Development 方法论（[Makr.io SDD Guide 2026](https://product.makr.io/learn/spec-driven-development)、[Claude Code SDD in Action](https://alexop.dev/posts/spec-driven-development-claude-code-in-action/)）。核心洞察：

- LLM 的幻觉/漂移主要不是模型问题，是**输入不够结构化**。
- 同一个"帮我发稿"的需求，裸 prompt 做 10 次有 10 种结果；带 schema 约束做 10 次，9 次收敛。
- 2026 年 AWS Bedrock、OpenAI、Anthropic 都把 structured output 作为一等公民（[AWS Bedrock Structured Outputs](https://aws.amazon.com/blogs/machine-learning/structured-outputs-on-amazon-bedrock-schema-compliant-ai-responses/)）——这是行业信号：**AI-Ready 输入 = 生产力下限的提升**。

### 1.2 AI-Ready 任务的六字段契约

每一个任务在进入池之前，必须能被描述成下面这个骨架（领域字段可扩展，但六个核心字段必填）：

```yaml
# AI-Ready Task Skeleton（核心字段，所有领域通用）
task_id: "mm-2026-05-09-001"          # 幂等键，重跑不重复
domain: "social-media"                 # 领域：social-media / github-dev / data-ops / research
intent: "publish-article"              # 意图动词+宾语，必须是枚举值
input_refs:                            # 领域输入的引用（不是内容，是指针）
  - type: "markdown"
    path: "./drafts/2026-05-09.md"
  - type: "image"
    path: "./assets/cover.png"
contract:                              # 验收契约（明确的完成定义）
  success_criteria:
    - "文章已发布到小红书，返回 URL"
    - "封面图已上传且被采纳"
  failure_modes:                       # 已知失败模式，触发人工
    - "登录态失效"
    - "平台风控拦截"
human_in_loop:                         # 人工介入策略
  mode: "none"                         # none / notify-after / approve-before
  reviewers: []                        # approve-before 时必填
notify:                                # 通知策略（task-2 要求）
  on_success: ["email:max@xxx.com"]
  on_failure: ["email:max@xxx.com"]
```

**这六个字段的刚性约束，就是 AI-Ready 的定义**。缺任何一个，拒绝入池。

### 1.3 领域字段扩展原则（task-2 要求：不做过多扩展）

遵循 YAGNI + 数据契约最小集原则：

| 领域 | 必须扩展的字段 | 不扩展什么 |
|---|---|---|
| `social-media` | `platform`（小红书/公众号/X）、`tone`、`tag_list` | 不做 A/B 测试字段，不做定时发布以外的调度 |
| `github-dev` | `repo`、`spec_file`（SDD 规范文件路径）、`test_command` | 不做 CI/CD 配置生成，不做多仓库联动 |
| `data-ops` | `source_uri`、`target_uri`、`transform_type` | 不做数据血缘追踪，不做质量监控 |
| `research` | `query`、`sources`、`output_format` | 不做引用网络、不做自动综述 |

**原则**：每个领域只保留"AI Agent 执行必需"的字段，不吸纳业务系统的全部字段。想扩展时问一句："不加这个字段，AI 能不能交付？" 能，就不加。

### 1.4 结构化输入三种载体（task-2 要求）

| 载体 | 适用场景 | 工具链 |
|---|---|---|
| **YAML** | 单任务、人写、可读性优先 | 直接手写 / VSCode snippet |
| **JSON** | 程序生成、API 接入、机器写 | Pydantic / Zod schema 校验 |
| **Excel / CSV** | 批量任务、非技术人员录入、行=任务 | pandas 读入 → 转 JSON 入池 |

三种载体必须**收敛到同一个内部 Schema**（1.2 的骨架），池里不保留原始格式差异。Excel 一行 = 一个任务 = 一份 JSON。

## 2. 任务池架构（五层，自下而上）

```
┌─────────────────────────────────────────────────────────────┐
│  L5  观测层   Langfuse / OpenTelemetry  成本 · 耗时 · 失败率    │
├─────────────────────────────────────────────────────────────┤
│  L4  执行层   Claude Code(SDD) │ n8n │ OpenClaw │ LangGraph   │
├─────────────────────────────────────────────────────────────┤
│  L3  路由层   router.py  根据 domain + human_in_loop 分通道    │
├─────────────────────────────────────────────────────────────┤
│  L2  任务池   SQLite / Redis Streams  幂等 · 重试 · 优先级      │
├─────────────────────────────────────────────────────────────┤
│  L1  入口层   YAML / JSON / Excel → Pydantic Schema 校验器     │
└─────────────────────────────────────────────────────────────┘
```

**每层的唯一职责**：

- **L1 入口层**：只做一件事——把三种载体转成 §1.2 的内部 Schema，不合规直接拒绝，返回具体错误行号。推荐 Pydantic v2（Python）或 Zod（TS）。
- **L2 任务池**：轻量化，SQLite + WAL 模式就够单机 10w 任务/天。不要上来就 Kafka——[Frugal.co 的 TaskOrchestrator 论述](https://frugal.co/blog/scaffolding-for-ai-the-frugal-way) 证明小规模场景下简单队列胜过复杂编排。
- **L3 路由层**：纯函数，输入 task，输出 `(channel, executor, config)`。不带状态，不带副作用。详见 §4。
- **L4 执行层**：**复用开源，不自己写**。每个 executor 是一个 adapter，约 50 行 Python 包住对应开源工具的 SDK/CLI。
- **L5 观测层**：Langfuse 跑 LLM 调用链追踪，Prometheus + Grafana 跑系统指标。2026 年的可观测性共识是 LLM 调用必须单独建模（[AI Agent Production Guide 2026](https://calmops.com/ai/ai-agent-production-deployment-2026/)）。

### 2.1 为什么是这五层，不多不少

| 层 | 能不能合并到上层 / 下层？ | 结论 |
|---|---|---|
| L1 | 合并到 L2？会让池里混杂校验失败的垃圾数据 | 独立 |
| L2 | 用内存队列行不行？进程崩溃任务就没了 | 独立，必须持久化 |
| L3 | 合到 L4？每个 executor 都写一遍路由逻辑，重复 | 独立 |
| L4 | 合到 L3？那就等于自己重写 n8n/Claude Code | 独立，专注 adapter |
| L5 | 合到 L4？每个 executor 自己打日志，没有全局视图 | 独立 |

## 3. 开源组件选型矩阵（不重复造轮子）

### 3.1 执行层选型决策树

```
任务 domain 是什么？
├── github-dev（代码开发/重构/PR）
│   └── ✅ Claude Code + cc-sdd 插件（SDD 规范驱动）
├── social-media / data-ops（跨 API 集成 · 多步编排）
│   └── ✅ n8n 自托管（400+ 节点 · 可视化 · 可暴露 webhook）
├── IM 交互 / 常驻 agent（Telegram/Discord/Slack 响应）
│   └── ✅ OpenClaw（本方案的下游 executor，不是竞品）
├── 复杂 DAG / 状态机 / 人工审核中断恢复
│   └── ✅ LangGraph（checkpoint · HITL 原生支持）
└── 定时/触发器编排
    └── ✅ Prefect 3.x 或 n8n 原生调度
```

### 3.2 2026 年各方案的真实适用边界（一句话点评）

参考 [开源 AI Agent 平台对比 2026](https://jimmysong.io/blog/open-source-ai-agent-workflow-comparison/)、[LangGraph vs n8n 实战](https://orangeloops.com/2025/06/building-ai-agents-with-langgraph-vs-n8n-a-hands-on-comparison/)、[CrewAI vs LangGraph vs n8n 决策树](https://codecoincognition.substack.com/p/ai-agent-frameworks-compared-crewai)：

| 工具 | 2026 最佳定位 | 用它的理由 | 不用它的理由 |
|---|---|---|---|
| **Claude Code + cc-sdd** | 代码类任务的"最后一公里"执行器 | SWE-bench 80.9% · 自带 Agent Skills · SDD 原生 | 非代码任务它干不了 |
| **n8n** | 跨 SaaS 工作流编排 | 400+ 节点 · 自托管 · 可视化调试 | 需要记忆/复杂状态时会翻车 |
| **OpenClaw** | IM 驻留型 agent 网关 | 跨 Telegram/Discord/Slack · 自托管隐私 | 不是开发工作流，是交互网关 |
| **LangGraph** | 需要 checkpoint + HITL 的复杂状态机 | 原生暂停/恢复 · Anthropic 生态 | prototype 阶段 50 行就开始复杂 |
| **Dify** | RAG + Agent 混合应用 | 中文社区活跃 · UI 友好 | 不是任务调度引擎 |
| **Prefect 3.x** | 数据工程背景的定时/批处理 | Pythonic · 成熟调度 | 对 LLM 支持不原生 |
| **Dagster** | 数据资产血缘驱动 | 数据血缘一等公民 | 学习成本高，过度设计 |

### 3.3 推荐组合（task-1 诉求）

**最小可用栈（MVP，2 周可上线）**：

```
L1  入口        Python + Pydantic v2           ~100 行
L2  任务池      SQLite + WAL                    ~150 行
L3  路由        纯函数 router                    ~80 行
L4  执行器      Claude Code CLI adapter         ~60 行
                n8n webhook adapter             ~60 行
L5  观测        Langfuse SDK（免费版即可）       ~30 行
L6  通知        SMTP / Resend API               ~40 行
─────────────────────────────────────────────────────
                                                ~520 行
```

**不写什么**：不写 DAG 编排引擎（用 n8n）、不写 LLM 调用层（用 Claude Code SDK）、不写 IM bot（用 OpenClaw）、不写日志存储（用 Langfuse）。这 520 行只做"胶水 + 契约校验 + 路由"，其他全部外包给成熟开源。

## 4. 人工介入三通道（解决木桶原理）

> 直接回应你的 background：**"像自媒体运营、github 工具开发等其实人工不参与，AI 完全可以交付。"**

### 4.1 三通道定义

按 `human_in_loop.mode` 字段路由到三条并行通道，互不阻塞：

| 通道 | `mode` 值 | 特征 | 典型任务 | SLA |
|---|---|---|---|---|
| **🚀 Auto 通道** | `none` | 全自动，成功/失败都只发邮件 | 发稿、抓取数据、跑 lint、翻译 | 秒级-分钟级 |
| **📬 Notify-After 通道** | `notify-after` | 自动执行，事后人工**抽查** | GitHub PR、报告生成、数据清洗 | 分钟级-小时级 |
| **🛑 Approve-Before 通道** | `approve-before` | 执行前必须人工审批 | 付款、对外发言、不可逆操作 | 按审批人响应时间 |

**关键设计**：三条通道**物理上是独立的 worker 进程池**，Auto 通道堵塞绝不影响其他。这就是"木桶不再是木桶"的机制——把桶拆成三个。

### 4.2 任务类型打标的默认规则（避免每次都让用户填）

路由器内置一个 intent → mode 的默认映射，用户可覆盖：

```python
# router/defaults.py
DEFAULT_MODE = {
    # Auto：低风险 · 可撤回 · 重跑成本低
    "social-media.draft":         "none",
    "social-media.publish":       "notify-after",    # 已发不可撤，抽查
    "github-dev.create-branch":   "none",
    "github-dev.open-pr":         "notify-after",    # PR 可关，但需要看看
    "data-ops.fetch":             "none",
    "research.summarize":         "none",

    # Approve-before：高风险 · 不可逆 · 对外
    "github-dev.merge-main":      "approve-before",
    "social-media.reply-comment": "approve-before",   # 对外发言
    "data-ops.delete":            "approve-before",
    "finance.transfer":           "approve-before",
}
```

### 4.3 抽查机制（Notify-After 通道的关键）

Notify-After 不是"不审核"，而是"**事后抽样审核 + 失败回滚**"：

- **成功任务**：随机 10% 发邮件摘要给审核人，其余只记录。
- **失败任务**：100% 发邮件 + 自动回滚（如撤回 PR / 删除草稿）。
- **连续失败 3 次**：自动降级为 Approve-Before，触发人工诊断。

这个策略参考 [Velocity Data Solutions - AI Orchestration Principles](https://www.velocitydatasolutions.net/insights/ai-orchestration-agent-handling/) 提出的"证据驱动的 agent 治理"——审核人不是流水线工人，是策略制定者。

### 4.4 为什么这是 2026 年的最佳实践

- **旧范式**：所有 AI 输出都人工过一遍 → 人工是瓶颈，AI 的价值被压缩到 10%。
- **2026 新范式**：**信任但验证**（Trust-but-Verify）—— 按任务风险分级，低风险全自动，高风险审批，中风险抽查。参考 [AI Agents in Production Guide 2026](https://neuralstackly.com/blog/ai-agents-production-2026-guide) 的生产部署章节。

## 5. 对接 AI Coding（Claude Code / Codex）做 SDD 开发

> 直接回应 task-1：**"交付可以让 AI-coding 直接进行 SDD 开发"**

### 5.1 `github-dev` 领域任务到 SDD 规范的转换

当一个 `domain: github-dev` 的任务入池，路由器会：

1. **生成 SDD 三件套文件**到 `./specs/{task_id}/`：
   - `requirements.md`（从 `contract.success_criteria` 展开）
   - `design.md`（Claude Code 首次运行时自动生成 + 人工 review）
   - `tasks.md`（设计拆分为可执行的 check-list）

2. **调用 Claude Code headless 模式**执行：

```bash
claude --print \
  --spec ./specs/{task_id}/ \
  --allowed-tools "Edit,Write,Bash(npm test:*),Bash(git *)" \
  --output-format json \
  "按 ./specs/{task_id}/tasks.md 逐项实现，每完成一项 commit 一次"
```

3. **验证契约**：跑 `contract.test_command`，通过才标记成功。

### 5.2 任务 YAML → SDD 规范的骨架映射

```yaml
# 输入（池里的 github-dev 任务）
task_id: "gh-2026-05-09-auth"
domain: "github-dev"
intent: "add-feature"
input_refs:
  - type: "repo"
    path: "./my-project"
contract:
  success_criteria:
    - "新增 /api/login 接口，JWT 鉴权"
    - "pytest 通过 · 覆盖率 >= 80%"
  test_command: "pytest --cov=. --cov-fail-under=80"
human_in_loop:
  mode: "notify-after"
```

```markdown
# 自动生成的 specs/gh-2026-05-09-auth/requirements.md
## 业务需求
新增 /api/login 接口，支持 JWT 鉴权

## 验收标准（EARS 格式）
- WHEN 用户携带正确的 username/password POST /api/login
  THEN 系统 SHALL 返回 200 + JWT token
- WHEN JWT 过期或无效 访问受保护接口
  THEN 系统 SHALL 返回 401

## 质量门
pytest 通过 · 覆盖率 >= 80%
```

### 5.3 推荐的 SDD 工具链（2026 成熟方案）

- **[cc-sdd](https://github.com/gotalab/cc-sdd)**（推荐）：最小可适配的 SDD harness，支持 Claude Code / Codex / Cursor / Copilot / Windsurf 多宿主，Agent Skills 原生。
- **[claude-code-spec-workflow](https://github.com/Pimzino/claude-code-spec-workflow)**（备选）：Requirements → Design → Tasks → Implementation 四阶段流水线，带 bug-fix 子流程。
- **[get-shit-done](https://github.com/gsd-build/get-shit-done)**（轻量替代）：meta-prompting + context engineering，适合不想重度依赖插件的场景。

**选型建议**：先用 cc-sdd，因为它是 spec-as-contract 的最小实现，跨工具可移植。如果团队已经深度使用 Claude Code 插件生态，用 manifoldlogic-sdd 或 mkelk-sdd ([Claude Plugin Hub](https://www.claudepluginhub.com/plugins/manifoldlogic-sdd-plugins-sdd))。

## 6. 邮件通知实现（task-2 必备功能）

### 6.1 最小实现（20 行 Python）

用 Resend（2026 年开发者邮件首选，免费额度 3k/月）或直接 SMTP：

```python
# notify/email.py
import resend
from jinja2 import Template

resend.api_key = os.environ["RESEND_API_KEY"]

TEMPLATE = Template("""
<h2>任务 {{ task_id }} {{ status_emoji }} {{ status }}</h2>
<p><b>Domain:</b> {{ domain }} / {{ intent }}</p>
<p><b>耗时:</b> {{ duration_s }}s · <b>Token:</b> {{ tokens }}</p>
{% if artifacts %}<p><b>产出:</b></p><ul>
{% for a in artifacts %}<li><a href="{{ a.url }}">{{ a.name }}</a></li>{% endfor %}
</ul>{% endif %}
{% if error %}<pre style="color:red">{{ error }}</pre>{% endif %}
<p><a href="{{ langfuse_trace_url }}">查看完整执行轨迹</a></p>
""")

def notify(task, result):
    for target in task.notify[f"on_{result.status}"]:
        if target.startswith("email:"):
            resend.Emails.send({
                "from": "agent@yourdomain.com",
                "to": target[6:],
                "subject": f"[{result.status.upper()}] {task.task_id}",
                "html": TEMPLATE.render(task=task, **result.__dict__),
            })
```

### 6.2 通知的三条铁律

1. **不要每个任务都发** —— Auto 通道默认**聚合日报**（每天 09:00 一封），只有失败才立即发。
2. **附可追溯链接** —— 每封邮件必须带 Langfuse trace URL，方便审核人点进去看全过程。
3. **带操作按钮**（进阶）—— Approve-Before 通道的邮件带"批准 / 拒绝"按钮，走签名链接回调到路由层，审核人不用登录系统。

### 6.3 其他可插拔通知后端

| 后端 | 场景 | 实现成本 |
|---|---|---|
| **Email (Resend/SMTP)** | 默认 · 异步 · 正式留痕 | ⭐ |
| **飞书 / 企微 webhook** | 团队实时响应 | ⭐ |
| **Telegram / Discord** | 个人 24h 响应（可复用 OpenClaw 通道） | ⭐⭐ |
| **Slack** | 海外团队 | ⭐ |

**保持可插拔**：`notify.on_success: ["email:x", "lark:group-id"]` 这种字符串前缀路由方式，新增通道只要加一个 adapter。

## 7. 与 OpenClaw 的差异化（task-3 正面回答）

### 7.1 一句话定性

> **本方案不是 OpenClaw 的竞品，而是它的"上游任务契约层"。**
> OpenClaw 解决 "agent 怎么 24/7 响应 IM 消息并动手执行"，本方案解决 "任务怎么被定义、分类、决定是否需要人工、最终路由到合适的执行器（其中一个可以是 OpenClaw）"。

参考：[Claude Code vs OpenClaw - MindStudio](https://www.mindstudio.ai/blog/claude-code-vs-openclaw)、[OpenClaw vs Claude API 层级对比](https://www.dench.com/blog/openclaw-vs-anthropic-claude-api)、[OpenClaw Architecture & Setup Guide 2026](https://vallettasoftware.com/blog/post/openclaw-2026-guide)。

### 7.2 九维对比表

| 维度 | **本方案（AI-Ready 任务池）** | **OpenClaw** |
|---|---|---|
| 核心抽象 | **任务（Task）** · 结构化 schema 驱动 | **消息（Message）** · IM 触发驱动 |
| 主要入口 | YAML / JSON / Excel 批量 | Telegram / Discord / Slack / WhatsApp |
| 核心能力 | 任务分类 + 路由 + 人工策略 | 跨 IM 常驻 + 工具调用 + 自主调度 |
| AI-Ready 输入 | ✅ 一等公民（六字段契约） | ❌ 自由文本，依赖 LLM 理解 |
| 人工介入分级 | ✅ 三通道（Auto/Notify/Approve） | ⚠️ 依赖用户在 IM 里手动干预 |
| 批量处理 | ✅ Excel 一次千行 | ❌ 面向对话，不擅长 |
| 领域可扩展 | ✅ 领域字段约束，YAGNI | ✅ 工具插件式 |
| 代码类 SDD | ✅ 原生对接 Claude Code | ⚠️ 通过调用 Claude Code 间接 |
| 部署形态 | 单进程 / 单机 SQLite | 自托管 gateway + LLM 后端 |
| 数据隐私 | 本地文件 / 本地 DB | 自托管（也是卖点） |
| **替代关系** | **不替代** | **被调用作为 IM executor** |

### 7.3 协同架构（不是二选一）

```
用户 Excel（批量任务）  ─┐
用户 YAML（单任务）     ─┼─▶ [本方案 L1-L3]  路由后分发：
Telegram 语音/文字      ─┘                     ├─ github-dev      → Claude Code
                                               ├─ social-media    → n8n
                                               ├─ im-response     → OpenClaw ★
                                               └─ approval-needed → 人工队列
```

**OpenClaw 在本架构里的角色**：当任务需要"通过 IM 和真人往返沟通"（比如客户 Q&A、审批通知后的对话），把这个子任务委派给 OpenClaw 当前端。本方案负责把任务放进去、把结果取出来。

### 7.4 什么时候该直接用 OpenClaw 而不是本方案

诚实地说：

- 你的工作流 **70% 以上都是 IM 响应 + 轻量任务**，不需要批量 + 不需要 SDD → 直接用 OpenClaw，别搞这一套。
- 你的工作流 **至少一半是批量任务 / 代码开发 / 跨域编排**，且人工审核是瓶颈 → 本方案 + OpenClaw 组合，不冲突。

## 9. 风险与边界

### 9.1 可预见的坑

| 风险 | 缓解 |
|---|---|
| LLM 偶发不按 schema 输出 | 用 Anthropic tool-use 强约束 + Pydantic 二次校验，失败进 DLQ |
| 跨 API 限流（小红书/GitHub） | 执行器层统一加 token bucket，超限自动排队 |
| Excel 列错位导致误入池 | L1 层做 header 指纹校验，字段名不一致直接拒绝 |
| 审核人不看邮件 | 7 天未响应的 Approve-Before 任务自动标记超时 + 升级通知 |
| 不可逆操作误触发（如 merge main） | 白名单 intent 列表，默认全部 Approve-Before，显式降级才放宽 |

### 9.2 刻意不做的事

- 不做多租户 / 权限系统：单人 / 小团队场景不需要。
- 不做可视化拖拽编排：有了就是又一个 n8n，不如直接用 n8n。
- 不做自研 LLM 调用层：Anthropic SDK / OpenAI SDK 已经够用，wrapper 是负资产。
- 不做跨语言 SDK：Python 单栈，跨语言调用走 HTTP/CLI。
